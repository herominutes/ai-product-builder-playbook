## Overview

Most product teams ship AI features without knowing what they cost to run. They discover the economics at scale — when it is too late to redesign the architecture, reprice the product, or explain to the CFO why gross margin dropped 12 points in a quarter.

> **Core Principle:** Every AI feature has a unit cost. If you do not model it before you build it, the market will model it for you — at the worst possible moment.

AI unit economics differ from traditional SaaS in one critical way: **marginal cost is not zero**. Every user interaction with an LLM costs money. Every inference has a price. Every token consumed is a line item. Product leaders with a financial background have a structural advantage here — and most of their peers do not.

## The AI Unit Economics Model

AI product economics operate across six layers. Each layer builds on the one above it. You cannot understand gross margin without understanding cost per user. You cannot understand cost per user without understanding cost per inference.

```
COST PER INFERENCE
  ↓ how much does one AI call cost?
COST PER USER
  ↓ how many calls does one user generate?
GROSS MARGIN BY FEATURE
  ↓ does this feature make money or destroy it?
LLM COST MODELLING
  ↓ which model, at what volume, at what price?
BUILD VS BUY VS FINE-TUNE
  ↓ what is the total cost of ownership?
BREAK-EVEN ANALYSIS
  ↓ when does this investment pay for itself?
```

| Layer | Core Question | Key Output |
|---|---|---|
| **Cost per inference** | What does one AI call cost? | $ per call |
| **Cost per user** | What does one user cost to serve per month? | $ per user / month |
| **Gross margin by feature** | Is this feature profitable at current pricing? | Feature gross margin % |
| **LLM cost modelling** | Which model and architecture optimises cost vs quality? | Model cost comparison |
| **Build vs buy vs fine-tune** | What is the true total cost of each path? | TCO comparison |
| **Break-even analysis** | When does this investment pay for itself? | Break-even month |

## Layer 1: Cost Per Inference

### 1.1 What Drives Inference Cost

Inference cost is a function of four variables:

```
COST PER INFERENCE FORMULA

Cost per inference = (Input tokens × Input price per 1K tokens)
                   + (Output tokens × Output price per 1K tokens)
                   + Infrastructure overhead (hosting, orchestration, retrieval)
```

| Cost Driver | Description | How to Estimate |
|---|---|---|
| **Input tokens** | Tokens sent to the model (prompt + context) | Avg prompt length × chars per token (≈ 4) |
| **Output tokens** | Tokens generated by the model | Avg response length × chars per token |
| **Model tier** | Price varies 10–100× across model sizes | Use provider pricing page |
| **Infrastructure** | Vector DB, API gateway, caching layer | Monthly infra cost ÷ monthly inference volume |
| **Retrieval cost** | Embedding generation, vector search | Embedding API cost + DB query cost |

### 1.2 Model Pricing Reference (order of magnitude — verify current pricing)

| Model Tier | Example | Input (per 1M tokens) | Output (per 1M tokens) | Best For |
|---|---|---|---|---|
| **Frontier** | GPT-4o, Claude Opus | $5–$15 | $15–$75 | Complex reasoning, high-stakes output |
| **Mid-tier** | Claude Sonnet, GPT-4o-mini | $0.15–$3 | $0.60–$12 | Balanced quality and cost |
| **Fast / cheap** | Claude Haiku, GPT-3.5 | $0.01–$0.25 | $0.03–$1.25 | High-volume, simple tasks |
| **Open source (hosted)** | Llama 3, Mistral | $0.10–$0.80 | $0.10–$0.80 | Cost-sensitive, privacy-required |
| **Self-hosted** | Any open model | Infrastructure cost only | Infrastructure cost only | High-volume at scale |

> **Rule of thumb:** Start with a mid-tier model. Only upgrade to frontier if quality benchmarks require it. A 10× price difference between tiers is common — always justify the upgrade with a measured quality delta.

## Layer 2: Cost Per User

### 2.1 From Inference to User Cost

Cost per inference becomes meaningful when multiplied by usage frequency. A $0.01 inference is cheap. A user who triggers 500 inferences per month costs $5.00 to serve — before infrastructure, support, or any other cost.

```
COST PER USER FORMULA

Monthly cost per user = Cost per inference
                      × Average inferences per user per month
                      + Fixed costs per user (infra, storage, support allocation)
```

### 2.2 Usage Pattern Segmentation

Do not average across all users. Usage is almost always distributed unevenly:

| User Segment | Typical Usage | Cost Profile |
|---|---|---|
| **Power users** | 5–10% of base, 50–80% of volume | High cost — monitor closely |
| **Regular users** | 30–40% of base, 15–35% of volume | Predictable — use for baseline |
| **Light users** | 50–60% of base, 5–15% of volume | Low cost — often unprofitable at low price points |
| **Churned / dormant** | Variable | Near-zero cost — do not include in active unit economics |

## Layer 3: Gross Margin by Feature

### 3.1 Feature-Level P&L

Most finance teams track gross margin at the product level. Product leaders should track it at the **feature level** — especially for AI features, where one expensive feature can drag down the entire product's margin.

```
FEATURE-LEVEL GROSS MARGIN

Feature revenue contribution:
  % of users who use this feature:              ___%
  Revenue attributable to feature:              $_______________

Feature cost:
  Total monthly inference cost (feature):       $_______________
  Infrastructure directly attributed:           $_______________
  Engineering maintenance (monthly amortised):  $_______________
  Total feature cost:                           $_______________

FEATURE GROSS PROFIT:   Revenue - Cost         = $_______________
FEATURE GROSS MARGIN:   Gross Profit ÷ Revenue = ___%
```

### 3.2 Margin Health Benchmarks

| Gross Margin | Assessment | Action |
|---|---|---|
| **> 70%** | Healthy for SaaS + AI | Maintain — monitor at scale |
| **50–70%** | Acceptable — watch closely | Optimise token usage and model tier |
| **30–50%** | Warning zone | Immediate cost review — reprice or re-architect |
| **< 30%** | Unsustainable | Stop new development — fix economics first |
| **Negative** | Destroying value | Urgent — consider gating or removing the feature |

## Layer 4: LLM Cost Modelling

### 4.1 Volume Sensitivity Analysis

LLM costs do not scale linearly with users because usage patterns vary. Model cost at 1,000 users is not simply 10× the cost at 100 users.

```
VOLUME SENSITIVITY MODEL

                Monthly Active Users
                100      1,000     10,000    100,000
                ------   -------   --------  ---------
Inferences/user   50       50        50         50
Total inferences  5,000    50,000    500,000    5,000,000

Model A (Frontier)
Cost per inf.    $0.050   $0.050    $0.045     $0.040   ← volume discount
Monthly cost     $250     $2,500    $22,500    $200,000

Model B (Mid-tier)
Cost per inf.    $0.005   $0.005    $0.004     $0.003
Monthly cost     $25      $250      $2,000     $15,000

Model C (Fast)
Cost per inf.    $0.001   $0.001    $0.001     $0.001
Monthly cost     $5       $50       $500       $5,000

Cost delta A vs C at 100K users: $195,000/month = $2.34M/year
```

> **Key insight:** Model selection is a pricing decision, not just a quality decision. At scale, the difference between a frontier and fast model can exceed the entire engineering payroll.

### 4.2 Token Optimisation Levers

Before switching models, exhaust these cost reduction levers:

| Lever | Typical Saving | Implementation Effort |
|---|---|---|
| **Prompt compression** | 20–40% input token reduction | Low — rewrite system prompts |
| **Output length limits** | 15–30% output token reduction | Low — add max_tokens parameter |
| **Caching frequent responses** | 30–60% inference reduction for repeated queries | Medium — implement semantic cache |
| **Model routing** | 40–70% cost reduction on simple tasks | Medium — route by task complexity |
| **RAG vs full context** | 50–80% input token reduction | High — requires retrieval architecture |
| **Batch processing** | 20–50% cost reduction on async tasks | Medium — shift to async where UX allows |
| **Fine-tuning** | 60–90% cost reduction post-training | High — requires training data and process |

### 4.3 Model Selection Decision Framework

```
MODEL SELECTION GUIDE

Step 1: Define the minimum acceptable quality threshold for this task.
        (What accuracy / quality score is required? Measure it.)

Step 2: Test the cheapest model first (Fast tier).
        Does it meet the threshold? → Use it.

Step 3: If not, test the mid-tier model.
        Does it meet the threshold? → Use it.

Step 4: If not, test the frontier model.
        Does it meet the threshold? → Use it, but immediately begin
        a fine-tuning programme to bring costs down.

Step 5: If no model meets the threshold at acceptable cost,
        the feature may not be viable today. Reassess in 6 months.

Never start with the frontier model and work down.
Always start with the cheapest model and work up.
```

## Layer 5: Build vs Buy vs Fine-Tune

### 5.1 Total Cost of Ownership Comparison

| Cost Category | Buy (API) | Build (Self-host) | Fine-Tune |
|---|---|---|---|
| **Upfront cost** | Low | High (GPU infra) | Medium (training run) |
| **Ongoing cost** | Per-token pricing | Infrastructure + ops | Reduced inference cost |
| **Latency** | Variable (shared infra) | Controlled | Controlled |
| **Data privacy** | Data leaves your infra | Data stays internal | Training data stays internal |
| **Quality control** | Provider-dependent | Full control | High control |
| **Maintenance** | Provider handles | Internal team required | Model refresh required |
| **Time to ship** | Days | Weeks–months | Weeks–months |

### 5.2 Decision Matrix

```
BUY vs BUILD vs FINE-TUNE DECISION

Answer each question:

1. Do you have data privacy requirements that prevent sending data to a third-party API?
   Yes → Self-host or fine-tune on internal infra

2. Is your monthly inference cost projected to exceed $50K within 12 months?
   Yes → Evaluate fine-tuning or self-hosting ROI

3. Do you have a labelled dataset of 1,000+ high-quality examples for your specific task?
   Yes → Fine-tuning is viable

4. Do you have an ML team capable of managing model infrastructure?
   No → Buy (API) until you do

5. Is your use case a commodity task (summarisation, classification, extraction)?
   Yes → Fine-tune a smaller model — frontier API is likely overkill

DECISION:
[ ] Buy (API)       — ship fast, validate, optimise later
[ ] Fine-tune       — specific task, have data, cost pressure is real
[ ] Self-host       — privacy requirements or very high volume
[ ] Hybrid          — buy for complex tasks, self-host for commodity tasks
```

## Anti-Patterns to Avoid

| ❌ Anti-Pattern | ✅ Better Approach |
|---|---|
| Starting with a frontier model and never revisiting | Start cheap, upgrade only when quality benchmarks require it |
| Tracking cost at the product level only | Track cost per feature — one expensive feature can hide behind blended margins |
| Ignoring power user economics | Segment usage — 10% of users often drive 70% of cost |
| Treating inference cost as fixed | Model pricing changes fast — review quarterly |
| Building without a break-even model | Every AI feature needs a break-even date before engineering starts |
| Fine-tuning without an ROI model | Fine-tuning has real upfront cost — model the payback period first |
| Optimising cost before validating quality | Never sacrifice quality to hit a cost target you set before measuring |
| Not caching repeatable queries | Semantic caching can cut inference cost 30–60% with moderate effort |

---

## Cadence

| Review Type | Frequency | Trigger |
|---|---|---|
| Inference cost check | Weekly | During initial launch period |
| Feature-level cost review | Monthly | Ongoing |
| Model pricing review | Quarterly | Provider pricing changes frequently |
| Full unit economics review | Quarterly | Before roadmap planning |
| Break-even reassessment | On major usage change | 2× or 0.5× projected usage |
| Build vs buy vs fine-tune review | Annually | Or when monthly cost exceeds $25K |

### Prompt 1 — Model the Unit Economics of an AI Feature

> **Best for:** Before building any AI feature — get a clear picture of what it will cost to run and whether the economics are viable.

```

3. Gross margin — compare cost to serve against my ARPU
4. Break-even — calculate how many users I need and how many months to payback

For each layer:
- Show the formula clearly
- Calculate the number from my inputs
- Flag any assumption I am making that could significantly change the result
- Tell me if the economics look healthy, marginal, or problematic at this layer before moving on

- The single highest-risk variable in my model
- The one optimisation lever that would have the biggest impact on my margin

My inputs:

Estimated input tokens per call: [number]
Estimated output tokens per call: [number]
Expected inferences per user per month: [number]
```
